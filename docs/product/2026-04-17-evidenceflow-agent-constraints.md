# 证据流 Agent Constraints

## 1. 商业与资源约束

- 付费模型仅允许使用 `DeepSeek 对话 API`。
- 默认不依赖任何其他必须付费的 LLM、搜索、向量或转写服务。
- 项目第一版必须能够在单机环境运行。
- 比赛版本优先保证成品感与稳定性，而不是大而全。

## 2. 技术约束

- 推荐采用 `Atom` 的能力边界作为底座参考，但不强制整仓硬融合。
- Embeddings 必须优先采用本地免费方案，例如 `fastembed` 或 `sentence-transformers`。
- 转写必须优先采用本地方案，例如 `faster-whisper` 或 `whisper.cpp`。
- 记忆与向量检索必须本地持久化，优先使用 `SQLite + LanceDB`。
- 外部检索应优先使用免费学术数据源，例如 `OpenAlex`、`arXiv`、`Semantic Scholar`。

## 3. 架构约束

- 第一版采用单租户、单工作区模型，不实现复杂权限系统。
- 前后端边界必须清晰，Agent 调度逻辑仅存在于后端。
- 所有外部能力通过 adapter 封装，避免业务代码直接依赖第三方接口。
- 不允许在核心流程中引入多个重型开源仓库直接耦合。

## 4. 产品约束

- 第一版聚焦“科研组会周进展推进”场景，不泛化到所有办公场景。
- 主流程必须聚焦：
  - 导入会议内容
  - 提取学生周进展、阻塞与风险
  - 捕获导师新 idea
  - 生成下周研究计划
  - 生成推荐阅读与验证指标
  - 生成交付物
- 第一版不做实时协同编辑。
- 第一版不做完整 PPT 编辑器，仅生成大纲或 Markdown/PDF 友好输出。
- 第一版不做实验自动执行、自动调参或训练平台集成。
- 第一版不把“证据核验”作为首页主卖点，而是作为辅助能力存在。

## 5. Agent 约束

- Agent 数量控制在 `1 个主控 + 4 个专职能力 Agent` 以内。
- 禁止把简单 prompt chaining 包装成“多 Agent 系统”。
- 每个 Agent 必须有明确输入、输出和失败回退策略。
- Agent 的主目标必须是推进“下一周研究执行”，而不是仅产出摘要。
- 证据相关 Agent 只在以下场景强制调用：
  - 导师明确要求补相关工作
  - 系统检测到结论或 idea 信息不足
  - 用户主动展开“参考依据”

## 6. 数据与安全约束

- 所有会议文本、转写结果、项目记忆默认本地存储。
- 学生历史进展、导师建议、实验失败记录默认按项目本地持久化。
- 不允许将原始密钥、录音、私有材料提交到仓库。
- 输出中不得伪造引用；无法找到来源时必须明确标记。
- 需要保留 transcript 到结论的可追溯链路。

## 7. 质量约束

- 所有结构化输出必须提供稳定 JSON schema。
- 所有核心功能必须配套单元测试或集成测试。
- 比赛 demo 必须准备固定样例数据，避免现场依赖不稳定外部服务。
- 推荐阅读和行动清单生成必须有 deterministic demo 样例，避免现场随机波动。
- 所有 `agent` 相关功能在交付前必须执行一次真实的 `DeepSeek` 在线验证，禁止仅依赖 mock 或假响应判断功能可用。
- 对 `DeepSeek` 的真实验证必须覆盖至少以下场景之一：
  - 周进展抽取
  - 导师 idea 捕获
  - 下周研究计划生成
  - 推荐阅读生成
  - 可选证据核验
- 后续开发中，如果某个 agent 功能没有完成真实 `DeepSeek` 请求验证，则该功能不能视为“已验证完成”。

## 8. 明确不做

- 多组织、多团队 SaaS 化管理
- 邮件、日历、IM 全量集成
- 企业客服或 CRM 工作流
- 端到端自动生成精美 PPT 页面
- 全自动后台提醒调度系统
- 首页级“结论正确性评分”产品形态
