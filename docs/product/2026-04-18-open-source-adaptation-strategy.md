# 证据流 Agent 开源项目魔改策略

## 1. 目的

本文件用于明确后续开发是否基于 GitHub 开源项目魔改，以及具体采用哪一个主底座仓库，避免后续开发偏向“从零重写”。

结论先行：

- 后续开发采用 `开源底座 + 定向裁剪 + 自研补齐` 的策略。
- 主底座仓库锁定为：`rush86999/atom`
- 不采用多仓库深度拼接，不做多个大型仓库的并行融合。

## 2. 为什么选择 Atom 作为主底座

### 2.1 与当前约束匹配

- 支持 `DeepSeek` 作为唯一付费模型提供方
- Embeddings 默认可走本地免费方案
- 本地优先、单机可运行
- 具备 agent orchestration、memory、workflow 这些我们需要的基本边界

### 2.2 与目标产品匹配

我们要做的是：

- 学生周进展提取
- 导师 idea 捕获
- 下周研究计划生成
- 推荐阅读生成
- 项目长期记忆

Atom 更接近“可扩展 agent 工作台”，比纯会议纪要产品更适合作为主底座。

### 2.3 为什么不直接选其他项目做主底座

- `Transcript Seeker`
  - 更像 transcript 交互产品
  - 适合作为 UI/交互参考，不适合作为总底座

- `Taskosaur`
  - 更像任务和项目管理系统
  - 适合作为行动计划与任务结构参考，不适合作为研究 agent 主底座

- `Paperlib`
  - 更像文献管理器
  - 适合作为推荐阅读和资料组织参考，不适合作为主流程底座

## 3. 魔改总策略

采用以下原则：

- `1 个主底座`
  - Atom
- `2-3 个参考仓库`
  - 只借设计、模块边界、交互思路
- `核心业务逻辑自研`
  - 学生周进展
  - 导师 idea
  - 下周研究计划
  - 推荐阅读

不做的事：

- 不把多个大型开源仓库代码硬拼进一个仓库
- 不复刻原始产品的完整功能
- 不为了“保留开源血统”而背上无关复杂度

## 4. 主底座保留内容

以下内容优先保留或借鉴 Atom 的思路：

- Agent orchestration 边界
- 本地优先配置方式
- provider 配置方式
- memory / storage 模型设计思路
- workflow 风格的任务推进逻辑

## 5. 主底座不保留内容

以下内容不应进入比赛版：

- 与 CRM / Slack / Gmail / Notion 等大规模集成相关模块
- 通用企业自动化场景
- Marketplace / 商业化扩展层
- 与当前科研组会场景无关的多代理复杂治理逻辑
- 所有与“企业业务自动化平台”强绑定的功能

## 6. 参考仓库与用途

### 6.1 `Taskosaur/Taskosaur`

用途：

- 参考 action plan 和任务管理结构
- 参考 conversational task execution 的任务落地方式

不直接复用：

- 完整 PM 系统
- Sprint / 多租户 / 看板系统

### 6.2 `Meeting-BaaS/transcript-seeker`

用途：

- 参考 transcript 时间线交互
- 参考“点 transcript 片段 -> 跳原文 / 原时间点”的体验

不直接复用：

- 会议 bot 接入
- 其外部 transcription API 依赖

### 6.3 `Future-Scholars/paperlib`

用途：

- 参考推荐阅读、资料组织、标签化、引用导出思路

不直接复用：

- 完整文献管理器
- Cloud sync / 插件生态

## 7. 需要自研的模块

以下模块不指望在开源底座里原样存在，必须自研：

- `weekly_progress_extractor`
  - 从组会中提取每个学生的本周进展

- `advisor_idea_capture`
  - 抽取导师的新想法、建议实验和验证目标

- `next_week_plan_generator`
  - 将 idea 转成可执行研究计划

- `reading_recommendation_service`
  - 根据当前 idea、阻塞点和项目背景生成阅读清单

- `student_research_memory`
  - 以学生为维度沉淀长期轨迹

## 8. 目录策略建议

后续若正式开始代码开发，建议保留一个明确的目录结构：

```text
frontend/    # 新前端工作台，允许借鉴 Atom 的信息结构，但不必照搬
backend/     # 新后端业务实现
references/  # 记录参考仓库信息、截图、字段映射、模块来源说明
docs/        # 产品、约束、实现计划
```

如果后续真的要直接拉 Atom 仓库代码，则建议：

- 将 Atom 作为 `upstream reference`
- 当前仓库作为比赛版实现仓库

而不是直接在 Atom 原仓库里堆改动。

## 9. 推荐开发姿势

后续开发按下面的顺序推进：

1. 固定开源底座策略
2. 建立最小项目骨架
3. 先实现主链路：
   - transcript -> 周进展
   - 周进展 + idea -> 下周计划
   - 下周计划 -> 推荐阅读
4. 再实现辅助链路：
   - claim -> evidence
   - transcript 回跳
5. 最后做 UI 打磨和 demo

## 10. 最终决策

后续如果继续由 Codex 开发，默认遵守以下约束：

- 这是一个 `基于 GitHub 开源项目能力边界进行魔改` 的项目
- 主底座是 `Atom`
- 核心业务逻辑不从别的项目硬拼，采用自研
- 参考仓库只作为模块和交互思路来源

这意味着：

- `不是完全从零重新发明`
- `也不是简单 fork 后改个皮`
- 而是：`以开源能力为基础，做一个场景明确、成本可控、适合比赛的垂直产品`
